## Опис завдання

На основі шостої лабораторної роботи оновити програму, додавши модуль детекції облич у реальному часі за допомогою глибоких нейромереж (DNN) та винести цей модуль в окремий фоновий потік.

Основні етапи роботи:

* Створення модульної архітектури детектора:
  * Реалізувати окремий клас `FaceDetector`.
  * Завантаження претренованої Caffe-моделі (ResNet-10 SSD).
* Багатопотокова обробка кадрів:
  * Ініціалізація фонового робочого потоку всередині `FaceDetector`.
  * Реалізація асинхронного циклу обробки, в якому потік очікує на новий кадр через `std::condition_variable`, не навантажуючи процесор і не блокуючи GUI-потік.
  * Забезпечення потокобезпечного обміну кадрами та результатами детекції за допомогою м'ютекса (`std::mutex`, `std::lock_guard`, `std::unique_lock`) і прапорця `std::atomic<bool>`.
* Додавання нового режиму обробки зображення — детекція облич (клавіша `F`).

## Опис архітектури

Програма поділена на класи, кожен у власних файлах `.hpp` та `.cpp`:

* `CameraProvider` — отримання кадрів з камери (`cv::VideoCapture`).
* `Display` — створення вікна та виведення кадрів (`cv::imshow`).
* `KeyProcessor` — обробка клавіатури та зберігання поточного режиму (`enum Mode`).
* `FrameProcessor` — обробка кадру залежно від режиму, малювання HUD та рамок облич.
* `MouseHandler` — обробка дій миші.
* `FaceDetector` — детекція облич у фоновому потоці.

Взаємодія потоків у `FaceDetector` побудована так. Головний потік методом `pushFrame()` під м'ютексом зберігає копію свіжого кадру та сповіщає робочий потік через `std::condition_variable`. Робочий потік у методі `run()` засинає на очікуванні умови й прокидається лише тоді, коли надходить новий кадр або програма завершується. Отримавши кадр, він виконує інференс мережі (`network_.forward()`) і під м'ютексом оновлює список знайдених облич. Головний потік методом `faces()` забирає останні координати й малює їх на поточному кадрі. Зупинку потоку контролює `std::atomic<bool> active_`.

Такий підхід через `condition_variable` не витрачає процесорний час на постійні перевірки: важка детекція не блокує цикл відображення, тому відео залишається плавним, а рамка може трохи відставати від різких рухів.

## Вимоги до системи та заліза

* ОС: Linux на базі Debian (тестувалося на Ubuntu; працює також на Kali Linux).
* Компілятор з підтримкою C++17 (g++), CMake, make.
* Бібліотека OpenCV з модулем `dnn` (`libopencv-dev`, версія 4.x).
* Камера: фізична вебкамера або віртуальна камера (наприклад DroidCam).

## Інструкція із запуску

### Linux (Ubuntu / Kali)

```bash
git clone <URL вашого репозиторію>
cd <папка репозиторію>/lab7
```

Перед запуском потрібно надати права на виконання скриптам:
```bash
chmod +x preinstall.sh build.sh run.sh
```
Після цього встановити залежності та завантажити модель детектора:
```bash
./preinstall.sh
```
Зібрати проєкт:
```bash
./build.sh
```
Запустити програму:
```bash
./run.sh
```

Скрипт `preinstall.sh` додатково завантажує два файли нейромережі у папку `assets`:
* `deploy.prototxt` — архітектура мережі;
* `res10_300x300_ssd_iter_140000.caffemodel` — ваги.

### Використання віртуальної камери DroidCam

DroidCam створює віртуальний відеопристрій (через модуль `v4l2loopback`), який зазвичай не є `/dev/video0`. Тому індекс камери можна передати аргументом скрипту запуску.

1. Запустіть DroidCam-клієнт та підключіть телефон.
2. Перегляньте список відеопристроїв:
```bash
v4l2-ctl --list-devices
```
3. Знайдіть пристрій DroidCam, подивіться його номер (наприклад `/dev/video2`) та запустіть програму з цим індексом:
```bash
./run.sh 2
```

Якщо аргумент не передано, використовується індекс `0`.
